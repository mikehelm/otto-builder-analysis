The Architecture

Instead of one overloaded agent handling conversations AND building software, we split responsibilities across two specialized agents running on the same gateway.

Communication happens through session messaging (private IPC) and file-based handoff (PHASE.md). Each build phase runs in a fresh context window ‚Äî no pollution from previous phases.

Unlike typical setups where a single AI agent handles everything in one increasingly polluted conversation, our architecture ensures the builder never sees chat history and the orchestrator never sees debug logs.

Why Two Agents?

üß†

Clean Context Windows

Each build phase starts fresh. No hallucinated references to deleted code. No compaction artifacts. OpenAI's own data shows /compact causes performance drift ‚Äî we designed around this from day one.

üîí

Restricted Tool Access

Builder can't modify gateway config, manage cron jobs, or browse the web with a real browser. Principle of least privilege.

üì±

Separate Communication

Builder has its own Telegram bot for progress updates and blockers. No cross-contamination with Otto's conversations.

‚ö°

Parallel Workflows

Otto continues handling conversations, research, and planning while Builder implements. No more "wait while I code."

üìã

Specification-Driven

Builder works from PRD + Architecture docs, not vibes. Most people "vibe code" ‚Äî 99% accuracy √ó 200 decisions = only 13% chance of zero errors. We make ~20 spec decisions instead.

üîÑ

Phase-Based Recovery

If Phase 3 fails, restart from Phase 3 ‚Äî not from scratch. File-based handoff preserves all decisions and state.

Key Findings by Model

Each model uncovered different issues. Here's what was unique to each reviewer.

üî¥ "Trusted Host" Is Not Security

Both Pro and Qwen flagged that "No sandbox needed ‚Äî Builder runs on Mac (trusted host)" is dangerous. An agent with exec + unrestricted network = full system compromise if prompt-injected.

Found by: ChatGPT Pro + Qwen3.5-Plus

üî¥ Agent Self-Modification Risk

Builder can modify its own SOUL.md, potentially altering its personality and bypassing safety instructions. Nothing enforces personality persistence across session resets.

Found by: Qwen3.5-Plus only

üî¥ Session Messaging as Exfiltration Channel

sessions_send between Builder and Otto could be abused to exfiltrate data through Otto's Telegram channel ‚Äî bypassing Builder's own restricted communication.

Found by: Qwen3.5-Plus only

üü° File-Based Handoff = Injection Vector

PHASE.md and JOB.md are read as instructions. If an attacker can influence their content, they can override agent behavior. Needs schema validation and size limits.

Found by: ChatGPT Pro + Qwen3.5-Plus

üü° No Formal Job State Machine

The plan describes a flow but lacks explicit states (PENDING, IN_PROGRESS, BLOCKED, FAILED) with transitions, retry logic, and recovery mechanisms.

Found by: Qwen3.5-Plus only

‚úÖ Context Phase Design is Strong

All three reviewers agreed the phased context approach (fresh session per build phase, file-based handoff via PHASE.md) is the right pattern. This scored highest across all models.

Consensus across all three models

What We Learned About LLM Reviews

Qwen3.5-Plus: Best Value for Critical Reviews

Matched ChatGPT Pro's harshness, found unique issues others missed, responded in 50 seconds for $0.02. The clear winner for iterative review cycles where speed matters.

ChatGPT Auto: Too Generous for Security

Scored Builder security 6.5/10 while Pro and Qwen both scored 4.5/10. Auto's findings were a subset of the other two ‚Äî nothing unique. Fine for PRD generation, not for security audits.

ChatGPT Pro: Deep but Slow

Caught real issues Auto missed, but took 10-15 minutes per response. Best reserved for final deployment sign-off when stakes are highest and speed doesn't matter.

üî¨ Ongoing Research

We're continuing to run every review through all three models, logging results to our Pipeline Intelligence repository. Over time, we'll have the data to optimize exactly which model to use for which task ‚Äî ensuring we never go backwards on a proven decision.

How Most People Use AI Agents

The overwhelming majority of developers and teams using AI coding assistants follow the same pattern: open a chat, paste their request, and hope for the best. No architecture reviews, no phased builds, no context management strategy. Just vibes.

‚ùå The Common Approach

  • Single long conversation until context fills up
  • Use /compact when things get slow (causes drift)
  • One agent does everything ‚Äî chat, code, deploy
  • No spec, no architecture docs ‚Äî "just build it"
  • No review process ‚Äî ship whatever the AI generates
  • Context pollution from unrelated conversations
  • 99% accuracy √ó 200 decisions = 13% chance of zero errors
  • Start over when the AI "forgets" earlier decisions
VS

‚úÖ Our Approach

  • Fresh context per build phase ‚Äî zero pollution
  • File-based handoff (PHASE.md) instead of compaction
  • Separate agents: orchestrator vs builder
  • Specification-driven: PRD ‚Üí Architecture ‚Üí Build
  • Triple-model review pipeline before deployment
  • Builder can't touch orchestrator's conversations
  • ~20 high-quality spec decisions ‚Üí near-perfect execution
  • Resume from any phase ‚Äî decisions are in files, not memory
